# .principles — Live Demo

> Demonstrated with **GitHub Copilot CLI** on [**Robocode Tank Royale**](https://github.com/robocode-dev/tank-royale) — a real project where `.principles` is already in use.

`.principles` ships three commands that form a complete workflow:

| Command | Purpose |
|---------|---------|
| `dot-scout` | Analyse a project, detect its stack, and create `.principles` files |
| `dot-audit` | Review code against activated principles — find, fix, commit, and open a PR |
| `dot-prime` | Activate principles before writing or editing code |

---

## 0 · Install

Before using any commands, install `.principles` into your project. This demo uses **Copilot** (CLI + IDE):

```
./install.sh copilot C:\Code\tank-royale
```

This installs the `dot-scout`, `dot-prime`, and `dot-audit` skills into the target project and vendors the principle catalog. See [INSTALL.md](../INSTALL.md) for the full guide (Claude Code, Codex, interactive mode, etc.).

## 1 · `dot-scout` — Detect & Configure

**What it does:** Scans your project tree, detects languages, frameworks, and domain signals, then places `.principles` files at the right directory levels — just like `.gitignore`, but for engineering guidance.

**Run it:**

```
> dot-scout
```

**Result** — `.principles` files appear throughout the project:

```

├── buildSrc/
│   └── ...
├── bot-api/
│   └── ...
├── gui-app/
│   └── ...
├── schema/
│   ├── schemas/
│   ├── scripts/
│   │   └── .principles            ◄ scripts
│   └── .principles                ◄ schema
├── server/

│   ├── build/
│   ├── docs/
│   ├── src/
│   ├── .principles                ◄ server (Kotlin)
│   ├── build.gradle.kts
│   ├── gradlew
│   ├── gradlew.bat
│   ├── jpackage.md
│   ├── jpackage-config.json
│   ├── r8-rules.pro

├── sounds/

├── .getattributes
├── .gitattributes
├── .gitignore
├── .principles                    ◄ root
└── AGENTS.md
```

Each `.principles` file activates the groups and individual principles relevant to that part of the tree. Subdirectories inherit from parents and can add, narrow, or suppress.

**Example** — the root `.principles` file generated by `dot-scout`:

```
# Generated by /scout
# Detected: Java/Kotlin, Python, TypeScript, C# / Multi-language bot APIs / Real-time game, WebSocket
# Last analysed: 2026-03-27

@java
@kotlin
@typescript
@python
@csharp
```

Each `@group` expands to a curated set of principles from the catalog. Subdirectory `.principles` files can layer on more specific groups (e.g. `@schema`, `@docs`) or suppress individual principles with `!ID`.

## 2 · `dot-audit recorder` — Audit, Fix & Ship

The full `dot-audit` workflow in one session: find issues → fix them → commit → push → open a PR.

**Run it:**

```
> dot-audit recorder
```

### Step 1 — Findings

The audit scans the `recorder` module against the active `.principles` hierarchy and reports 10 findings grouped by severity:

---

**Audit complete — 10 findings.**

🔴 **High:**

- `.../cli/Recorder.kt:32` **[CODE-CC-SYNC-SHARED-STATE]** — `recordingObserver` is written on main thread and read on stdin thread without `@Volatile` or synchronization, violating JVM visibility guarantees. → Declare `@Volatile var recordingObserver: RecordingObserver? = null`.

🟠 **Medium:**

- `.../cli/Recorder.kt:43` **[CODE-CC-TASK-BASED-CONCURRENCY]** — Raw `Thread {}` construction bypasses lifecycle management. → Replace with `Executors.newSingleThreadExecutor` with a named daemon thread factory.
- `.../cli/Recorder.kt:42` **[CODE-CC-STRUCTURED-CONCURRENCY]** — Fire-and-forget daemon thread has no parent scope; failures are silently lost. → Hold the `ExecutorService`, call `shutdownNow()` in a `finally` block in `run()`.
- `.../core/RecordingObserver.kt:106` **[CODE-CS-FAIL-FAST]** — `catch (_: Exception) { // ignore }` silently discards I/O errors from `close()`. → Replace with `catch (e: Exception) { log.warn("...", e) }`.
- `.../core/RecordingObserver.kt:124` **[EFFECTIVE-JAVA-MINIMIZE-ACCESSIBILITY]** — `handleServerHandshake` is public but only called from private `processMessage`. → Add `private` modifier.
- `.../cli/Recorder.kt:32` **[CODE-TP-TYPE-STATE-MACHINES]** — `lateinit var` with 4× `isInitialized` guards; uninitialized state should be typed as null. → Replace with `@Volatile var recordingObserver: RecordingObserver? = null` and null-safe access.

🔵 **Low:**

- `.../cli/Recorder.kt:73` **[CODE-DX-NAMING]** — `exitProcess(1)` used for a deliberate user quit; exit code 1 conventionally signals error. → Change to `exitProcess(0)`.
- `.../cli/Recorder.kt:3` **[CODE-CS-YAGNI]** — `CliktCommand`, `option`, `versionOption`, and `Version` are imported but unused in `RecorderRuntime`. → Remove the four unused imports.
- `.../core/RecordingObserver.kt:18` **[EFFECTIVE-JAVA-WRITE-DOC-COMMENTS]** — `RecordingObserver`, `RecorderRuntime`, `RecorderCli`, and their public methods have no KDoc. → Add KDoc covering contract, parameters, return values, and thread-safety.
- `.../cli/Recorder.kt:14` **[CODE-CC-DOCUMENT-THREAD-SAFETY]** — `RecorderRuntime` uses two threads sharing mutable state with no thread-safety documentation. → Add class-level KDoc describing the two-thread model and `@Volatile` visibility guarantee.

> `handleServerHandshake` was Kotlin-public by default but is only ever called from the private `processMessage`. Changed to `private`.
>
> 🟠 **MEDIUM — `lateinit var` + `isInitialized` guards** (CODE-TP-TYPE-STATE-MACHINES)
>
> Four `isInitialized` runtime introspection checks deferred type safety to runtime. Replaced with `@volatile var recordingObserver: RecordingObserver? = null` and null-safe local-val captures.
>
> 🔵 **LOW — `exitProcess(1)` for normal quit** (CODE-DX-NAMING)
>
> Exit code 1 conventionally signals failure; a deliberate user-initiated quit should use exit code 0. Changed to `exitProcess(0)`.
>
> 🔵 **LOW — Unused imports** (CODE-CS-YAGNI)
>
> `CliktCommand`, `option`, `versionOption`, and `Version` were imported but unused. Removed.
>
> 🔵 **LOW — Missing KDoc** (EFFECTIVE-JAVA-WRITE-DOC-COMMENTS)
>
> `RecordingObserver`, `RecorderRuntime`, `RecorderCli`, and all their public methods lacked documentation. KDoc added to all.
>
> 🔵 **LOW — Thread-safety undocumented** (CODE-CC-DOCUMENT-THREAD-SAFETY)

## 3 · `dot-prime` — Activate Before Coding

**What it does:** Loads the resolved `.principles` hierarchy and distills it into a compact set of 5–10 rules that the AI agent keeps in its active context while writing or editing code.

Unlike `dot-scout` (which writes files) and `dot-audit` (which produces visible findings and fixes), `dot-prime` works *behind the scenes* — it shapes how the AI thinks, not what it outputs. That makes it hard to demo visually, but it's arguably the most impactful command for day-to-day coding.

**Run it:**

```
> dot-prime
```

**Sample output:**

```
Active rules:
- CODE-CS-DRY: Give every piece of knowledge one authoritative home.
- CODE-CS-KISS: Prefer the simplest solution that works.
- CODE-DX-NAMING: Name things by what they represent, not how they're stored.
- SOLID-SRP: Give each module exactly one reason to change.
- CODE-CS-FAIL-FAST: Detect errors immediately and report them loudly.
- CODE-CC-SYNC-SHARED-STATE: Guard every cross-thread field with synchronization.
- EFFECTIVE-JAVA-MINIMIZE-ACCESSIBILITY: Make each class and member as inaccessible as possible.
- CODE-CS-YAGNI: Delete code you don't need today.

Apply these rules to all code you generate. Proceed.
```

From this point on, every line of code the AI writes or reviews is filtered through these principles — no extra prompting needed.

### When to use it

| Scenario | Command |
|----------|---------|
| Starting a new coding task | `dot-prime` — loads principles for the current directory |
| Working on a specific file | `dot-prime src/server/Recorder.kt` — tailored to that file's position in the hierarchy |
| Overriding with specific groups | `dot-prime @security-focused @kotlin` — explicit principle selection |
| Quick focused review | `dot-prime CODE-CC-SYNC-SHARED-STATE CODE-CS-FAIL-FAST` — just these two principles |

---

## The Full Workflow

```
dot-scout          →  analyse project, create .principles files
dot-prime          →  activate principles before coding
dot-audit <target> →  review, fix, commit, and ship
```

> `.principles` doesn't teach the AI — the AI already knows SOLID, OWASP, DDD, and the rest.
> It gives the AI **your intent**: which principles matter **here**, in **this** part of **your** codebase.
