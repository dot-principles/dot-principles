The format is based on [Keep a Changelog](https://keepachangelog.com/), and this project adheres to [Semantic Versioning](https://semver.org/).

> **Note:** `.principles` is in its early days ‚Äî this is an experimental release. The system is being actively used and iterated on, but expect rough edges, evolving formats, and breaking changes as things stabilize.

## [v0.9.0] ‚Äî 2026-03-31

### Changed

- **Consistent frontmatter** ‚Äî all generated skill/prompt files now carry the same core fields (`description`, `argument-hint`, `allowed-tools`, `version`, `authors`) regardless of tool. Tool-specific extras (`name`, `license`, `mode`) added by each wrapper template.
- **Single frontmatter block** ‚Äî Copilot CLI skills and IDE prompts no longer have double `---` frontmatter; one unified block per file.
- **`install.sh` rewritten** (~700 ‚Üí ~770 lines) ‚Äî template-driven generation, `--list` shows review status, `--help` colorized with BOLD/DIM.
- **`uninstall.sh` colors** ‚Äî added YELLOW and DIM; help text uses BOLD for sub-commands and DIM for category labels; `NEUTRAL` marker upgraded from plain `-` to dim-colored.
- **`/scout` Phase 6.0** ‚Äî reads `install.cfg` (if present) as authoritative source for enabled review targets before falling back to file-based detection.
- **`.gitattributes`** ‚Äî added `*.cfg text eol=lf` to prevent CRLF issues in template config files.
- **DESIGN.md** ‚Äî new "Template System" section (¬ß9) with schema docs and "Adding a New AI Tool" guide; updated installer targets table.
- **INSTALL.md** ‚Äî documents `copilot-cli`, `copilot-ide`, interactive mode, and `--list`; split Copilot section into CLI + IDE.

## [v0.7.0] ‚Äî 2026-03-27

### Added

- **Gated fix-to-PR workflow** (Phases 8‚Äì10) ‚Äî after the read-only review (Phases 1‚Äì7), `/audit` now offers three optional gated phases that handle fix, commit, and PR creation. Each phase is a mandatory approval checkpoint ‚Äî the default is always to stop and ask:
  - **Phase 8 ‚Äî Fix** ‚Äî asks "Would you like me to fix these findings?"; on approval creates a `fix-<target-slug>` branch, applies every finding's `fix` field, and runs existing tests.
  - **Phase 9 ‚Äî Commit** ‚Äî presents the commit message and PR body for review, then offers three choices: *commit only*, *commit and push*, or *exit*.
  - **Phase 10 ‚Äî Pull Request** ‚Äî asks "Shall I open a pull request?"; on approval creates a PR targeting the default branch with a structured body (summary, per-finding rationale, changes table).
- **Structured commit message and PR body formats** ‚Äî gated workflow produces a `fix(<target>): resolve <N> audit findings (<severities>)` commit message with per-finding line items, and a PR body with severity-grouped rationale sections and a changes table.

### Changed

- **Strict state-machine semantics** ‚Äî identifying issues ‚âÝ permission to fix; fixing ‚âÝ permission to commit; committing ‚âÝ permission to push or open a PR. Silence, hints, context, or likely intent do not count as approval. Phases cannot be skipped, combined, or inferred.

## [v0.6.0] ‚Äî 2026-03-24

### Added

- **`**Summary:**` field** ‚Äî all 374 principle files now include a `**Summary:**` field: a one-line, actionable rule appearing after `**Applies-to:**`. Used in the compiled block and required for all new contributions (see [CONTRIBUTING.md](CONTRIBUTING.md)).
- **Compact principle index** (`index.tsv`) ‚Äî `install.sh vendor` generates a pipe-delimited flat file (`ID|LAYER|SUMMARY`) of all 373 principles at `.principles-catalog/index.tsv`; `/scout` reads this single file to build the compiled block in one pass, eliminating per-namespace file walking.
- **Compiled block** ‚Äî `/scout` Phase 6 compiles all active principles into a compact `<!-- .principles: begin --> ‚Ä¶ <!-- .principles: end -->` block and injects it into three targets:
  - `.claude/rules/principles.md` ‚Äî created if absent; Claude Code reads this directory automatically
  - `AGENTS.md` ‚Äî three-case handling: hub layout (`.ai/principles.md` created + table row added), simple layout (block injected directly), absent (file created from scratch)
  - `.github/copilot-instructions.md` ‚Äî injected unless the file is a pointer/redirect file
- **`/audit` fast path** ‚Äî reads compiled block (tier 1) instead of tree-walking `.principles` files; loads `.context-audit.md` per namespace (tier 2) for full guidance.
- **`/prime` fast path** ‚Äî same fast path: compiled block as tier 1, `.context-prime.md` per namespace as tier 2.
- **`vendor` subcommand** ‚Äî `install.sh vendor <dir>` copies the catalog subset referenced by the project's `.principles` files into `<dir>/.principles-catalog/`. Commit this directory so the compiled block and fast paths work without the full catalog repo present.
- **AGENTS.md as cross-agent injection target** ‚Äî AGENTS.md is now a first-class injection target for the compiled block, enabling any agent that reads AGENTS.md (OpenAI Codex, Claude Code, Copilot, etc.) to receive the active principle set automatically.

### Changed

- **Repo-only install** ‚Äî `install.sh` now requires a `<dir>` argument; global install (without `<dir>`) is no longer supported. The primary install command is `./install.sh all <project-dir>`.
- **No more `~/.principles`** ‚Äî principle data is no longer copied to a global `~/.principles` directory. Commands reference `.principles-catalog/` inside the project. The `{{PRINCIPLES_DIRECTORY}}` placeholder now resolves to the project-local `.principles-catalog/`.

### Removed

- **Global install** ‚Äî `./install.sh claude`, `./install.sh copilot`, and `./install.sh all` without a `<dir>` argument are no longer supported.
- **`~/.principles` data directory** ‚Äî removed from the install/uninstall flow.
- **Cursor support** ‚Äî Cursor is no longer a supported install target; the `.cursor/rules/principles.mdc` target has been dropped.

### Fixed

- **Uninstall** ‚Äî now strips compiled blocks from `.claude/rules/principles.md`, `.ai/principles.md`, `AGENTS.md`, and `CLAUDE.md`; removes `.principles-catalog/`; removes legacy `~/.principles` if present.

## [v0.3.2] ‚Äî 2026-03-22

### Changed

- **`/audit` explicit principle override** ‚Äî `/audit` now accepts an explicit principle spec to force a specific principle set, bypassing `.principles` files and dynamic detection entirely. Three equivalent syntaxes are supported:
  - `<spec> on <target>` ‚Äî natural language: `/audit DDD on src/orders`
  - `<target> --with <spec>` ‚Äî flag syntax: `/audit src/orders --with DDD`
  - `@<group> <target>` ‚Äî group-prefix syntax: `/audit @ddd src/orders`
  - Multiple groups supported comma-separated: `/audit clean-arch, solid on src/`
  - `principle_source` in `audit-output.json` reports `explicit: <spec>` when override is active.
