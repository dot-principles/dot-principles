# SEC-ARCH-SUPPLY-CHAIN-SECURITY — Verify provenance and integrity across the build chain

**Layer:** 2 (contextual)
**Categories:** security, architecture, devops
**Applies-to:** all
**Summary:** Verify artifact provenance by digest, generate SBOMs, sign releases, and use hermetic build pipelines.

## Principle

Treat every external dependency, container base image, build tool, and CI action as a potential attack vector. Verify artifact integrity through cryptographic signatures and provenance attestations; generate and publish a Software Bill of Materials (SBOM) for every release; and structure build pipelines to be hermetically sealed and reproducible so that every released artifact can be traced to reviewed source code through a tamper-resistant chain of custody.

## Why it matters

Supply chain attacks — SolarWinds, XZ Utils, event-stream — demonstrate that a single compromised dependency or build tool can cascade into thousands of downstream organisations simultaneously. Lock files and checksum verification are necessary but insufficient: they verify what was fetched, but not whether the upstream source was itself compromised. Provenance attestations and hermetic builds provide a higher guarantee: every binary can be traced back to specific reviewed source code, built by a specific pipeline step, without external network access during the build. An attacker who compromises a dependency but cannot compromise the build attestation cannot inject malicious code into a verifiably attested artifact.

## Violations to detect

- No SBOM generated or published for released artifacts
- Container base images referenced by mutable tag (`FROM ubuntu:22.04`) rather than by immutable digest (`FROM ubuntu@sha256:...`)
- Build pipelines that fetch dependencies from the public internet at build time without a private artifact proxy or vendored copy
- Third-party CI actions (GitHub Actions, Tekton tasks) referenced by a floating branch or tag rather than a pinned commit SHA
- No cryptographic signature verification of downloaded artifacts before they are used in the build
- Build pipeline steps with broad outbound network access during compilation, where a compromised dependency could exfiltrate secrets

## Inspection

- `grep -rnE 'uses:\s+[^@]+@(main|master|latest|v[0-9]+)$' --include="*.yml" --include="*.yaml" $TARGET` | HIGH | GitHub Actions pinned to mutable ref
- `grep -rnE '^FROM [a-zA-Z0-9_./:-]+:[a-zA-Z0-9_.-]+$' --include="Dockerfile" --include="*.dockerfile" $TARGET` | MEDIUM | Docker image pinned by tag not digest

## Good practice

- Generate an SBOM (CycloneDX or SPDX format) on every CI build and publish it alongside the release artifact
- Pin all container base images to their `sha256` digest; automate digest updates through a bot (Dependabot, Renovate) so they remain current
- Pin all third-party CI actions to a specific commit SHA; use a tool such as `pin-github-actions` to manage them
- Use a hermetic build environment: no outbound network access during compilation; all dependencies are resolved from a private proxy or vendored into the repository
- Sign released artifacts and their SBOMs using Sigstore/Cosign; publish provenance attestations following the SLSA framework
- Implement SLSA Level 2 as a minimum: hosted build platform, version-controlled build scripts, and provenance generated by the build service
